# 动态世界书管理脚本

## 🎯 功能说明

这个脚本会根据游戏上下文（当前区域、境界、出现的人物等）**自动启用或禁用**世界书条目，从而：

- ✅ **节省 token**：只激活当前需要的条目
- ✅ **优化性能**：减少无关信息对 AI 的干扰
- ✅ **提升体验**：让 AI 更专注于当前场景

## 📖 工作原理

### 蓝灯 vs 绿灯

在酒馆的世界书中，条目有两种激活模式：

- **🔵 蓝灯（Constant）**：无条件激活，只要条目启用就会发送给 AI
- **🟢 绿灯（Selective）**：条件激活，需要在消息中匹配到关键词才激活

本脚本的策略是：
1. **需要的条目** → 设为蓝灯（立即激活）
2. **不需要的条目** → 设为绿灯（等待关键词触发）

### 判断逻辑

脚本会检查：
- ✅ 当前所在区域（从 MVU 变量中读取）
- ✅ 当前境界等级
- ✅ 红颜列表中的角色
- ✅ 最近 5 条消息的内容
- ✅ 其他 MVU 变量数据

## 🔧 配置方法

### 1. 基础配置

编辑 [`index.ts`](index.ts:16) 中的 CONFIG：

```typescript
const CONFIG = {
    // 要管理的世界书名称
    WORLDBOOK_NAME: 'current',  // 'current' = 当前角色卡的主世界书
    
    // 是否启用调试日志
    DEBUG: true,
    
    // 上下文窗口（检查最近几条消息）
    CONTEXT_WINDOW: 5,
};
```

### 2. 自定义规则

在 [`index.ts:42`](index.ts:42) 的 `RULES` 数组中添加你的规则：

```typescript
{
    // 条目名称匹配（支持字符串或正则）
    entryPattern: /^\[地图\]\s*神州/,
    
    // 启用条件
    shouldEnable: ctx => ctx.currentRegion === '神州',
    
    // 优先级（可选，数字越大越优先）
    priority: 10,
}
```

## 📝 规则示例

### 示例1：地图条目

```typescript
{
    entryPattern: /^\[地图\]\s*魔域/,
    shouldEnable: ctx => ctx.currentRegion === '魔域',
    priority: 10,
}
```

**效果**：只有当主角在魔域时，`[地图] 魔域` 条目才会设为蓝灯。

### 示例2：人物条目

```typescript
{
    entryPattern: /^\[角色\]\s*孟十七/,
    shouldEnable: ctx => {
        // 在红颜列表中，或最近消息提到她
        return (
            ctx.mvuData?.红颜?.['孟十七'] !== undefined ||
            ctx.recentMessages.some(msg => msg.includes('孟十七'))
        );
    },
    priority: 8,
}
```

**效果**：孟十七在红颜列表中，或最近有人提到她时，才激活。

### 示例3：境界条目

```typescript
{
    entryPattern: /^\[境界\]\s*练气/,
    shouldEnable: ctx => {
        const level = ctx.mvuData?.本尊?.等级 ?? 1;
        return level >= 1 && level <= 4;
    },
    priority: 7,
}
```

**效果**：只有在练气期（等级 1-4）时才激活练气相关条目。

### 示例4：系统条目（始终启用）

```typescript
{
    entryPattern: /^\[系统\]/,
    shouldEnable: () => true,  // 始终返回 true
    priority: 100,
}
```

**效果**：系统相关条目永远保持蓝灯。

## 🎮 使用方法

### 1. 部署脚本

```bash
pnpm build
```

在酒馆助手的"脚本库"中添加：
```
dist/美化状态栏/脚本/动态世界书管理/index.js
```

勾选"自动加载"。

### 2. 设置世界书条目命名规范

为了让脚本能够识别条目，建议使用统一的命名规范：

- `[地图] 区域名` - 地图相关
- `[角色] 角色名` - 人物相关
- `[境界] 境界名` - 境界说明
- `[事件] 事件名` - 特殊事件
- `[系统] 功能名` - 系统设定（始终启用）
    
### 3. 查看日志

打开浏览器开发者工具（F12），查看 Console：

```

[动态世界书] ✅ 启用（蓝灯）: [地图] 神州
[动态世界书] ❌ 改为绿灯: [地图] 魔域
[动态世界书] 更新完成: 启用蓝灯 3 个, 改为绿灯 5 个
```

## 💡 Context 上下文对象

在规则的 `shouldEnable` 函数中，你可以访问以下上下文信息：

```typescript
interface Context {
    /** 当前所在区域 */
    currentRegion: string;
    
    /** 最近消息内容（数组） */
    recentMessages: string[];
    
    /** MVU 变量数据（完整的角色卡数据） */
    mvuData: {
        本尊: { 等级, 修为, 灵根, 体质, 行踪, ... },
        红颜: { [名字]: {...} },
        NPC图鉴: { [名字]: {...} },
        世界时钟: {...},
        世界地图: {...},
        ...
    } | null;
    
    /** 当前聊天 ID */
    chatId: string;
}
```

### 常用判断模式

```typescript
// 判断区域
ctx.currentRegion === '魔域'

// 判断境界
ctx.mvuData?.本尊?.等级 >= 5

// 判断是否在红颜列表
ctx.mvuData?.红颜?.['孟十七'] !== undefined

// 判断最近消息是否提到
ctx.recentMessages.some(msg => msg.includes('掌门'))

// 判断体质
ctx.mvuData?.本尊?.体质 === '太虚龙剑体'

// 判断灵石数量
(ctx.mvuData?.本尊?.灵石 ?? 0) > 100
```

## ⚡ 性能优化建议

### 优先级设置

```typescript
priority: 100  // 系统级（始终启用）
priority: 10   // 地图级（高优先级）
priority: 7    // 境界级（中优先级）
priority: 5    // 人物级（一般优先级）
priority: 3    // 事件级（低优先级）
```

### 规则顺序

脚本会按照优先级从高到低检查规则，一旦找到匹配的规则就停止。因此：

1. 把最常用的规则设置更高的优先级
2. 把系统相关的规则放在最前面
3. 把特殊情况的规则放在一般规则之前

## 🐛 故障排除

### 条目没有被管理

**原因**：条目名称不匹配任何规则

**解决**：
1. 检查条目名称格式
2. 检查规则中的 `entryPattern` 是否正确
3. 开启 `DEBUG: true` 查看日志

### 脚本不生效

**原因**：脚本未正确加载或世界书未绑定

**解决**：
1. 检查 Console 是否有 `[动态世界书] 脚本已加载`
2. 确认角色卡绑定了主世界书
3. 确认脚本 URL 正确且可访问

### Token 没有节省

**原因**：太多条目被设为蓝灯

**解决**：
1. 检查规则的 `shouldEnable` 条件是否太宽松
2. 考虑将一些条目改为仅在提到时启用
3. 优化世界书条目数量和内容

## 📚 最佳实践

1. **分类清晰**：使用统一的命名前缀（如 `[地图]`、`[角色]`）
2. **按需激活**：只在真正需要时才设为蓝灯
3. **保留绿灯**：让 AI 有能力通过关键词激活相关内容
4. **定期检查**：通过日志了解哪些条目被频繁激活
5. **渐进优化**：先设置基础规则，再根据实际效果调整

## 🎯 节省效果示例

**不使用动态管理**：

**使用动态管理**：

- **节省约 80-90% 的世界书 token！**
